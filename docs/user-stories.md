# OmniVOC - ユーザーストーリー

## ロール定義

| ロール | 説明 |
|--------|------|
| エンドユーザー | フィードバックを送る人（各プロダクトの利用者） |
| プロダクト開発者 | 自分のプロダクトに OmniVOC を導入する人 |
| 管理者 | OmniVOC ダッシュボードでフィードバックを確認・集約・チケット化する人 |

---

## Phase 0: 導入

### US-0a: プロジェクト登録

> プロダクト開発者として、OmniVOC にプロジェクトを登録して project_key を取得したい。
> それにより、自分のプロダクトからフィードバックを送信できるようになる。

**受け入れ条件:**
- ダッシュボードからプロジェクトを登録できる
- project_key が自動発行される
- 導入方法（API エンドポイント・ウィジェットタグ）が表示される

### US-0b: Web ウィジェットの埋め込み

> プロダクト開発者として、`<script>` タグ 1 行で Web サイトにフィードバックウィジェットを設置したい。
> それにより、最小限の実装でユーザーの声を集められる。

**受け入れ条件:**
- `<script src="https://omnivoc.vercel.app/widget.js" data-project-key="xxx">` を貼るだけで動く
- フローティングボタンが表示され、クリックするとフォームが開く
- 送信すると OmniVOC API にフィードバックが送られる

### US-0c: 既存 Bot からの API 連携

> プロダクト開発者として、既存の Slack Bot / LINE Bot から OmniVOC API を叩いてフィードバックを転送したい。
> それにより、新しい Bot を作らずに既存チャネルからユーザーの声を集められる。

**受け入れ条件:**
- `POST /api/feedback` に project_key・channel・content を送ると受け付けられる
- 201 レスポンスが返る
- 認証は project_key で行う

## Phase 1: 収集（貯める）

### US-1: Web ウィジェットからフィードバック送信

> エンドユーザーとして、Web サイト上のフローティングウィジェットからフィードバックを送信したい。
> それにより、思いついた瞬間にすぐ声を届けられる。

**受け入れ条件:**
- フローティングボタンをクリックするとフォームが開く
- テキストを入力して送信できる
- 送信後「ありがとうございます」のフィードバックが表示される
- DB にチャネル種別 `web` として保存される

### US-2: 既存 Slack Bot 経由でフィードバック収集

> エンドユーザーとして、普段使っている Slack Bot にフィードバックを送りたい。
> それにより、業務ツールから離れずに声を届けられる。

**受け入れ条件:**
- 各プロダクトの既存 Slack Bot がフィードバックを検知して OmniVOC API に転送する
- DB にチャネル種別 `slack`、project_key 付きで保存される

### US-3: 既存 LINE Bot 経由でフィードバック収集

> エンドユーザーとして、普段使っている LINE Bot にフィードバックを送りたい。
> それにより、スマホから手軽に声を届けられる。

**受け入れ条件:**
- 各プロダクトの既存 LINE Bot がフィードバックを検知して OmniVOC API に転送する
- DB にチャネル種別 `line`、project_key 付きで保存される

## Phase 2: 集約 → チケット化

### US-4: フィードバック一覧の閲覧

> 管理者として、ダッシュボードで全プロジェクト・全チャネルのフィードバックを一覧で見たい。
> それにより、ユーザーの声の全体像を把握できる。

**受け入れ条件:**
- 全フィードバックが時系列で一覧表示される
- プロジェクトでフィルタできる
- チャネル種別でフィルタできる
- ステータス（new / ticketed / resolved）でフィルタできる

### US-5: フィードバックから GitHub Issue を作成

> 管理者として、フィードバックを選んで GitHub Issue を作成したい。
> それにより、ユーザーの声を開発タスクとして追跡できる。

**受け入れ条件:**
- フィードバック詳細画面に「チケット化」ボタンがある
- 対象リポジトリを選択できる
- Issue タイトル・本文を編集してから作成できる（元のフィードバック内容がプリセットされる）
- 作成後、フィードバックのステータスが `ticketed` に変わる
- フィードバックと Issue URL が紐付けて記録される

### US-6: 複数フィードバックをまとめてチケット化

> 管理者として、似た内容のフィードバックを複数選択してまとめて 1 つの Issue にしたい。
> それにより、同じ要望を 1 つのチケットで追跡できる。

**受け入れ条件:**
- 一覧画面で複数のフィードバックを選択できる
- 「まとめてチケット化」で 1 つの Issue が作成される
- 選択した全フィードバックが同一 Issue に紐付けられる
- 全フィードバックのステータスが `ticketed` に変わる

---

## Phase 3: 後続フィードバックの処理

### US-7: 新規フィードバックを既存チケットに紐付け

> 管理者として、新しいフィードバックを既に存在する Issue に紐付けたい。
> それにより、同じ要望の件数を追跡でき、優先度判断の材料になる。

**受け入れ条件:**
- フィードバック詳細画面に「既存チケットに紐付け」ボタンがある
- 既存 Issue を検索・選択できる
- 紐付け後、フィードバックのステータスが `ticketed` に変わる

### US-8: 対応済みフィードバックのスルー

> 管理者として、既に対応済み（Issue がクローズ済み）の内容と同じフィードバックを「解決済み」にマークしたい。
> それにより、対応不要なものを効率的にさばける。

**受け入れ条件:**
- フィードバック詳細画面に「解決済みにする」ボタンがある
- 紐付け先の Issue（任意）を指定できる
- ステータスが `resolved` に変わる

## 将来（V2）: AI 補助

### US-9: 類似フィードバックの自動グルーピング提案

> 管理者として、AI が類似フィードバックをグルーピング提案してくれると嬉しい。
> それにより、大量のフィードバックを効率的にさばける。

### US-10: 既存チケットとの自動マッチング提案

> 管理者として、新しいフィードバックが入った時に「この Issue と同じ内容では？」とサジェストしてほしい。
> それにより、見落としや重複チケットを防げる。
