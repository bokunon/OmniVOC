# OmniVOC - ユーザーストーリー

## ロール定義

| ロール | 説明 |
|--------|------|
| エンドユーザー | フィードバックを送る人（各プロダクトの利用者） |
| プロダクト開発者 | 自分のプロダクトに OmniVOC を導入する人 |
| 管理者 | OmniVOC ダッシュボードで AI 判定を確認・修正する人 |

---

## Phase 0: 導入

### US-0a: プロジェクト登録

> プロダクト開発者として、OmniVOC にプロジェクトを登録して project_key を取得したい。
> それにより、自分のプロダクトからフィードバックを送信できるようになる。

**受け入れ条件:**
- ダッシュボードからプロジェクトを登録できる
- project_key が自動発行される
- 導入方法（API エンドポイント・ウィジェットタグ）が表示される

### US-0b: Web ウィジェットの埋め込み

> プロダクト開発者として、`<script>` タグ 1 行で Web サイトにフィードバックウィジェットを設置したい。
> それにより、最小限の実装でユーザーの声を集められる。

**受け入れ条件:**
- `<script src="https://omnivoc.vercel.app/widget.js" data-project-key="xxx">` を貼るだけで動く
- フローティングボタンが表示され、クリックするとフォームが開く
- 送信すると OmniVOC API にフィードバックが送られる

### US-0c: 既存 Bot からの API 連携

> プロダクト開発者として、既存の Slack Bot / LINE Bot から OmniVOC API を叩いてフィードバックを転送したい。
> それにより、新しい Bot を作らずに既存チャネルからユーザーの声を集められる。

**受け入れ条件:**
- `POST /api/feedback` に project_key・channel・content を送ると受け付けられる
- 201 レスポンスが返る
- 認証は project_key で行う

## Phase 1: 収集 + AI 自動判定

### US-1: Web ウィジェットからフィードバック送信

> エンドユーザーとして、Web サイト上のフローティングウィジェットからフィードバックを送信したい。
> それにより、思いついた瞬間にすぐ声を届けられる。

**受け入れ条件:**
- フローティングボタンをクリックするとフォームが開く
- テキストを入力して送信できる
- 送信後「ありがとうございます」のフィードバックが表示される
- DB にチャネル種別 `web` として保存される
- AI が自動で分類する（auto_new / auto_linked / auto_resolved）

### US-2: 既存 Slack Bot 経由でフィードバック収集

> エンドユーザーとして、普段使っている Slack Bot にフィードバックを送りたい。
> それにより、業務ツールから離れずに声を届けられる。

**受け入れ条件:**
- 各プロダクトの既存 Slack Bot がフィードバックを検知して OmniVOC API に転送する
- DB にチャネル種別 `slack`、project_key 付きで保存される
- AI が自動で分類する

### US-3: 既存 LINE Bot 経由でフィードバック収集

> エンドユーザーとして、普段使っている LINE Bot にフィードバックを送りたい。
> それにより、スマホから手軽に声を届けられる。

**受け入れ条件:**
- 各プロダクトの既存 LINE Bot がフィードバックを検知して OmniVOC API に転送する
- DB にチャネル種別 `line`、project_key 付きで保存される
- AI が自動で分類する

### US-4: フィードバックの AI 自動分類

> 管理者として、フィードバック受信時に AI が自動で分類してほしい。
> それにより、手動トリアージなしでフィードバックが整理される。

**受け入れ条件:**
- フィードバック受信時に AI が既存 Issue との類似度を判定する
- 類似する open Issue がある → `auto_linked`（suggested_issue_url を記録）
- 類似する closed Issue がある → `auto_resolved`（suggested_issue_url を記録）
- どちらにも該当しない → `auto_new`
- AI の判定理由（confidence score + 理由テキスト）が記録される

## Phase 2: 管理者による確認・チケット化

### US-5: AI 判定の確認キュー

> 管理者として、AI が分類したフィードバックを確認キューで一覧したい。
> それにより、AI の判定精度を検証しながら効率的にさばける。

**受け入れ条件:**
- AI 判定済み（auto_*）のフィードバックが確認キューに並ぶ
- AI の判定結果（分類・提案先 Issue・理由）が表示される
- プロジェクト・チャネル・ステータスでフィルタできる
- 承認 / 修正がワンクリックでできる

### US-6: AI 判定の承認

> 管理者として、AI の判定が正しければワンクリックで承認したい。
> それにより、確認作業を最速でさばける。

**受け入れ条件:**
- `auto_linked` → 承認で `ticketed`（提案された Issue に紐付け確定）
- `auto_resolved` → 承認で `resolved`（スルー確定）
- `auto_new` → そのまま確認済みとしてマーク

### US-7: AI 判定の修正

> 管理者として、AI の判定が間違っていた場合に修正したい。
> それにより、誤判定を正しい状態に直せる。

**受け入れ条件:**
- `auto_linked` → 別の Issue に変更して紐付け / 「類似ではない」で auto_new に戻す
- `auto_resolved` → 「同一ではない」で auto_new に戻す
- `auto_new` → 既存 Issue に紐付け変更

### US-8: フィードバックから GitHub Issue を作成

> 管理者として、`auto_new` のフィードバックを GitHub Issue にチケット化したい。
> それにより、ユーザーの声を開発タスクとして追跡できる。

**受け入れ条件:**
- 「チケット化」ボタンがある
- 対象リポジトリを選択できる
- Issue タイトル・本文を編集してから作成できる（元のフィードバック内容がプリセット）
- 作成後 `ticketed` に変わり、Issue URL が紐付けられる

### US-9: 複数フィードバックをまとめてチケット化

> 管理者として、似た内容のフィードバックを複数選択してまとめて 1 つの Issue にしたい。
> それにより、同じ要望を 1 つのチケットで追跡できる。

**受け入れ条件:**
- 一覧画面で複数のフィードバックを選択できる
- 「まとめてチケット化」で 1 つの Issue が作成される
- 選択した全フィードバックが同一 Issue に紐付けられ `ticketed` に変わる

## 将来: 段階的な自動化

### US-10: 高信頼度の判定を自動承認

> 管理者として、AI の信頼度が十分高い判定は自動承認してほしい。
> それにより、確認が必要な件数を減らし、本当に判断が必要なものに集中できる。

**受け入れ条件:**
- 信頼度スコアの閾値を設定できる
- 閾値以上のものは自動で承認される（ticketed / resolved に遷移）
- 自動承認されたものはログに記録され、後から確認可能
