# OmniVOC - フィードバック処理フロー

## 全体の流れ

```
[各プロダクト]                              [OmniVOC]

Web: <script> ウィジェット ─────┐
Slack Bot (既存): API 転送 ─────┤
LINE Bot (既存): API 転送 ──────┤──→ POST /api/feedback ──→ DB に蓄積
その他: API 転送 ───────────────┘              │
                                              ▼
                                        AI が自動判定
                                              │
                                    ┌─────────┼─────────┐
                                    ▼         ▼         ▼
                              新規トピック  既存に類似   対応済みと同一
                                    │         │              │
                                    ▼         ▼              ▼
                             auto_new   auto_linked    auto_resolved
                                    │         │              │
                                    └─────────┴──────────────┘
                                              │
                                              ▼
                                    管理者がダッシュボードで確認
                                    （AI 判定の承認 or 修正）
```

## 設計方針

**手動トリアージはしない。** フィードバック受信時に AI が自動で分類・紐付け・スルー判定を行う。
管理者の役割は AI 判定の **確認・修正** に集中する。

特に立ち上げ初期は AI 判定の精度を検証するため、全件確認を前提とする。

## フィードバックのライフサイクル

```
┌──────────────────────────────┐
│     フィードバック受信         │
│     POST /api/feedback        │
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│     AI 自動判定               │
│                              │
│  1. 既存 Issue と類似か？     │
│  2. 対応済み Issue と同一か？  │
│  3. 新規トピックか？          │
└──────────────┬───────────────┘
               │
     ┌─────────┼──────────┐
     ▼         ▼          ▼
 auto_new  auto_linked  auto_resolved
     │         │          │
     │    (AI が既存    (AI が対応済み
     │    Issue を提案)  Issue を提案)
     │         │          │
     └─────────┴──────────┘
               │
               ▼
┌──────────────────────────────┐
│   管理者がダッシュボードで確認  │
│                              │
│   ・AI 判定が正しい → 承認    │
│   ・AI 判定が違う → 修正     │
│   ・新規をチケット化 → Issue  │
└──────────────┬───────────────┘
               │
     ┌─────────┼──────────┐
     ▼         ▼          ▼
  ticketed  ticketed   resolved
  (新規     (既存       (スルー)
   Issue)    Issue紐付)
```

## ステータス定義

| ステータス | 意味 | 設定タイミング |
|-----------|------|---------------|
| `auto_new` | AI 判定: 新規トピック | フィードバック受信時に AI が自動設定 |
| `auto_linked` | AI 判定: 既存 Issue に類似 | フィードバック受信時に AI が自動設定 |
| `auto_resolved` | AI 判定: 対応済み Issue と同一 | フィードバック受信時に AI が自動設定 |
| `ticketed` | 管理者が確認済み・チケット化済み | 管理者が承認 or 新規 Issue 作成時 |
| `resolved` | 管理者が確認済み・対応不要 | 管理者が承認時 |

### ステータス遷移

```
auto_new       → ticketed  (管理者がチケット化)
               → resolved  (管理者が対応不要と判断)

auto_linked    → ticketed  (管理者が紐付けを承認)
               → ticketed  (管理者が別の Issue に修正して紐付け)
               → auto_new  (管理者が「類似ではない」と判断 → 新規扱いに戻す)

auto_resolved  → resolved  (管理者がスルーを承認)
               → auto_new  (管理者が「同一ではない」と判断 → 新規扱いに戻す)
```

## AI 判定の仕組み

### 判定ロジック

```
フィードバック受信
       │
       ▼
既存の open Issue と類似度を計算
       │
       ├── 類似度が高い Issue あり → auto_linked（suggested_issue_url を記録）
       │
       ▼
既存の closed Issue と類似度を計算
       │
       ├── 類似度が高い Issue あり → auto_resolved（suggested_issue_url を記録）
       │
       ▼
どれにも該当しない → auto_new
```

### 類似度計算（候補）

| 方式 | メリット | デメリット |
|------|---------|-----------|
| Embedding + コサイン類似度 | 意味的な類似性を捉えられる | Embedding の生成・保存が必要 |
| LLM に直接判定させる | 柔軟・文脈理解が深い | コスト・レイテンシが高い |
| ハイブリッド（Embedding で候補絞り → LLM で最終判定） | バランスが良い | 実装が複雑 |

初期は **LLM 直接判定**でスタートし、量が増えたらハイブリッドに移行する想定。

## 管理者のダッシュボード体験

### 確認キュー

ダッシュボードには AI 判定済みのフィードバックが並ぶ。管理者は各件に対して:

- **承認**: AI の判定をそのまま採用（ワンクリック）
- **修正**: 別の Issue に紐付け直す / ステータスを変更する
- **チケット化**: `auto_new` のものを GitHub Issue にする

### 初期の運用

立ち上げ初期は **全件確認** を前提とする。AI の判定精度が安定してきたら:

1. 信頼度スコアが高いものは自動承認
2. 信頼度が低いものだけ管理者が確認

という段階的な自動化を検討する。
